# 我用手机调度一堆 Claude Code：工作方式发生了点“质变”

最近我把自己的开发协作方式，改成了一个很怪但很顺的组合：

- **OpenClaw（Codex 5.3）**：在手机上接收/发起任务、做调度
- **Claude Code**：真正动手改代码
- **tmux**：让每个任务变成一个随时可接管的后台会话

结果是：我不再开十几个窗口“盯进度”，而是像在管理一组 **jobs**。

## 1）以前的问题：窗口越开越多，但心智带宽越来越少
并行做事的时候（修 bug、改 UI、写文档、跑脚本），我经常陷入这种状态：

- VS Code 窗口开到 6 个
- 终端 tab 开到 12 个
- AI 聊天窗口再开几个

最烦的不是“乱”，而是：
- 我得在脑子里维护“哪个窗口在跑什么”
- 任务卡住了也不一定第一时间发现
- 最后一句 “done” 经常等于 “我也不知道改了啥”

## 2）新的工作流：把任务变成可调度、可观测、可接管
我现在更像在跑一个轻量的“调度系统”：

### 发起（手机也行）
我只要发一句话，例如：
> 把某个页面的按钮状态统一到设计系统，补齐加载态和错误态。

### 运行（每个任务一个 tmux session）
OpenClaw 会把任务启动成一个独立会话：
- 你可以把它理解为：**每个任务都有自己的“后台进程 + 日志”**
- 想看进度就 attach 进去看输出
- 卡住了可以直接接管键盘继续处理

### 交付（不是“done”，而是报告）
我强制每个任务结束前都要产出一个小报告：
- 改了哪些文件
- diff 统计
- lint/build 是否通过
- 风险评估、是否有 scope drift

这点很关键：**结束不再是聊天说完成，而是可审计的交付物**。

## 3）我说的“质变”是什么
它不是“AI 更聪明了”这种质变，而是工作方式的结构变了：

- 从管理窗口 → 管理作业（jobs）
- 从猜进度 → 随时可见（输出就是最朴素的 observability）
- 从不可接管 → 随时接管（tmux attach）
- 从一句 done → 一份可复盘的报告

最真实的感受是：并行推进时，我的注意力不再被“现场管理”吞噬。

## 4）什么时候值得试试
如果你也属于这种状态：
- 同时推进多个小任务
- 经常被上下文切换折磨
- 想要“随时接管 + 可复盘”的 AI 协作

那我很推荐你用类似方式把 AI 变成“可调度的工程作业”，而不是“多开几个聊天窗口”。

最后一句话总结：
**窗口变少了，但可控性变强了。**

---

### 可直接照抄的 3 条实践建议
1) **任务=一个 tmux session**：别开新窗口，给 session 起名。
2) **强制交付：diff + lint/build + 报告**：把验证写进流程。
3) **卡住就 attach 接管**：把“重开/重述上下文”降到最低。
